Key insight: RLS is your security boundary (who can do what). Sync rules/streams are your efficiency boundary (who gets what). A secure app needs both — they are complementary, not interchangeable.

Sync Rules (Legacy) vs Sync Streams

Sync Streams (edition 3) replace the legacy Sync Rules approach. Both control which data syncs to which device — but the mental model, syntax, and capabilities differ significantly.

Architecture comparison
Sync Rules (Legacy) · edition 1-2

You explicitly define buckets, each with parameter queries and data queries. The server pre-computes all possible bucket instances.

bucket_definitions: user_profile: # 1. Parameter query: who gets this bucket? parameters: SELECT request.user_id() AS profile_id # 2. Data queries: what data goes in it? data: - SELECT * FROM profile WHERE id = bucket.profile_id - SELECT * FROM quest WHERE bucket.profile_id in download_profiles

All data syncs upfront when the client connects. No way for the client to request specific data on-demand.

Client code:
// Connect and wait — all bucket data downloads await db.connect(connector); // Data is just... there. No subscription API. const quests = await db.getAll('SELECT * FROM quest');
Sync Streams · edition 3

You define streams with SQL queries. Clients subscribe to streams with parameters. Buckets are created implicitly.

streams: my_quests: # Single query — parameters inline query: | SELECT * FROM quest WHERE auth.user_id() in download_profiles auto_subscribe: true project_data: # On-demand: client subscribes with project_id query: | SELECT * FROM quest WHERE project_id = subscription.parameter('project_id') AND auth.user_id() in download_profiles

On-demand or auto — subscribe when needed, data syncs just-in-time with built-in TTL caching.

Client code:
// Subscribe on-demand with parameters + TTL const sub = await db .syncStream('project_data', { project_id: id }) .subscribe({ ttl: 3600 }); await sub.waitForFirstSync(); // Later: sub.unsubscribe() — data cached for TTL

How RLS and Sync Rules/Streams interact

RLS and sync filtering are applied at completely different points in the data pipeline. Understanding when each fires is critical.

Write path: RLS enforced, sync rules ignored
Read/sync path: Sync rules enforced, RLS not consulted
Common misconception: "If my RLS allows a user to SELECT a row, that user will also get it via sync." False. RLS and sync filtering are independent. A row must match both the sync rules/streams query AND any RLS SELECT policy to be accessible through their respective paths. But they are evaluated in completely separate contexts.
Side-by-side: RLS policy vs sync query for the same table
RLS Policy · vote table

Controls who can write votes via Supabase API / RPC:

-- INSERT: only the creator CREATE POLICY "Enable vote insert only by creator" ON vote FOR INSERT TO authenticated WITH CHECK ( creator_id = auth.uid() ); -- UPDATE: only the creator CREATE POLICY "Enable vote update only by vote creator" ON vote FOR UPDATE TO authenticated USING (creator_id = auth.uid()) WITH CHECK (creator_id = auth.uid());

Scope: write operations only. Prevents impersonation and vote manipulation.

Sync Rule · vote table

Controls which votes sync to the device:

# Sync Rules (legacy) bucket_definitions: user_profile: parameters: SELECT request.user_id() AS profile_id data: - SELECT * FROM vote WHERE bucket.profile_id in download_profiles # Sync Streams (equivalent) streams: project_votes: auto_subscribe: true query: SELECT * FROM vote WHERE auth.user_id() in download_profiles

Scope: download sync only. User gets votes from all projects they downloaded.

Notice the pattern: RLS checks auth.uid() against creator_id (ownership). Sync rules check user_id against download_profiles[] (membership / subscription). Same table, completely different access patterns.
