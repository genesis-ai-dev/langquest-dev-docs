-- Primary table chain (the spine of the database)

The project-quest-asset-vote series of tables forms the spine of user-generated language data in the app. Projects, quests, assets and votes are all user-generated.

CREATE TABLE public.project (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  name text NOT NULL,
  description text,
  target_language_id uuid,
  active boolean DEFAULT true,
  creator_id uuid,
  private boolean DEFAULT false,
  visible boolean DEFAULT true,
  download_profiles ARRAY,
  ingest_batch_id uuid,
  priority smallint DEFAULT '0'::smallint,
  template text DEFAULT 'unstructured'::text CHECK ((template = ANY (ARRAY['unstructured'::text, 'bible'::text, 'fia'::text])) OR template IS NULL),
  versification_template text,
  CONSTRAINT project_pkey PRIMARY KEY (id),
  CONSTRAINT project_creator_id_fkey FOREIGN KEY (creator_id) REFERENCES public.profile(id)
);

Projects are created by the user and represent an effort by one or more people to translate some set of content from and/or into a particular language. 
A project is mainly linked to members (see membership subschema), content, and a language (see languoid subschema). It also has adjustable settings for if it is active (people can contribute to it), visible (people can see it), and private (only members can contribute to it).
For now, it also has a template associated with it to indicate what kind of template it uses. This will likely be expanded in the future so that projects (or maybe quests instead) will be linked to different template records. This idea is still in development.

CREATE TABLE public.quest (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  name text,
  description text,
  project_id uuid NOT NULL,
  active boolean NOT NULL DEFAULT true,
  creator_id uuid,
  visible boolean DEFAULT true,
  download_profiles ARRAY,
  ingest_batch_id uuid,
  parent_id uuid,
  metadata text,
  CONSTRAINT quest_pkey PRIMARY KEY (id),
  CONSTRAINT quests_project_id_fkey FOREIGN KEY (project_id) REFERENCES public.project(id),
  CONSTRAINT quest_creator_id_fkey FOREIGN KEY (creator_id) REFERENCES public.profile(id),
  CONSTRAINT quest_parent_id_fkey FOREIGN KEY (parent_id) REFERENCES public.quest(id)
);

A quest is like a folder - an organizational tool. In unstructured/custom template projects, users define the quests. In templated projects (like Bible or FIA for now), quests are predefined. E.g., a Bible project has book quests at the root level, and chapter quests within those book quests. 
FIA is similar, but the book quests hold pericopes (passages) instead of chapters. 
Quests can contain assets (kind of like files) or other quests. At this point there is no nesting limit. Similar to projects, quests have settings for active and visible (but not private).
For now, metadata holds information about which part of a template the quest belongs to (if relevant).

CREATE TABLE public.quest_asset_link (
  quest_id uuid NOT NULL,
  asset_id uuid NOT NULL,
  active boolean NOT NULL DEFAULT true,
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  download_profiles ARRAY,
  visible boolean NOT NULL DEFAULT true,
  ingest_batch_id uuid,
  CONSTRAINT quest_asset_link_pkey PRIMARY KEY (quest_id, asset_id),
  CONSTRAINT quest_assets_asset_id_fkey FOREIGN KEY (asset_id) REFERENCES public.asset(id),
  CONSTRAINT quest_assets_quest_id_fkey FOREIGN KEY (quest_id) REFERENCES public.quest(id)
);

The quest_asset_link table was originally created as a junction table to allow one quest to link to many assets it contains, and to allow one asset to link to multiple quests (where overlapping passages may have the same verse in multiple passage quests).
After realizing this overlapping situation wasn't common, we realized we could still use this junction table to allow one asset to be linked to multiple versions of the same quest.
Versions of a quest are a thing because of our append-only approach to allowing end users to contribute. This means User A and B can submit their own versions of the same templated quest (e.g., two versions of Mark chapter 2).
With only two version of the same quest, this junction table isn't needed yet. It's needed when we allow end users to create "remixes" - picking and choosing which assets from which versions will go into a new remixed version. This approach prevents duplication of assets.

CREATE TABLE public.asset (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  name text,
  source_language_id uuid,
  images ARRAY,
  active boolean NOT NULL DEFAULT true,
  creator_id uuid,
  visible boolean DEFAULT true,
  download_profiles ARRAY,
  ingest_batch_id uuid,
  project_id uuid,
  parent_id uuid,
  source_asset_id uuid,
  order_index integer DEFAULT 0,
  content_type text DEFAULT 'source'::text CHECK (content_type = ANY (ARRAY['source'::text, 'translation'::text, 'transcription'::text])),
  metadata text,
  CONSTRAINT asset_pkey PRIMARY KEY (id),
  CONSTRAINT asset_creator_id_fkey FOREIGN KEY (creator_id) REFERENCES public.profile(id),
  CONSTRAINT asset_parent_id_fkey FOREIGN KEY (parent_id) REFERENCES public.asset(id),
  CONSTRAINT asset_source_asset_id_fkey FOREIGN KEY (source_asset_id) REFERENCES public.asset(id),
  CONSTRAINT asset_project_id_fkey FOREIGN KEY (project_id) REFERENCES public.project(id)
);

Assets represent a small translatable unit of language information. 
In our context, this usually means a single Bible verse, but it could also be a line from a poem, a single word, a series of images, a recording, or some combination of those things. 
But all the modalities (audio, image, text) all represent the same piece of information (e.g., the same verse).
For example, when end users are recording individual audio clips in the recording interface, each recording ends up becoming an asset. 
End users can then open that asset up and submit translations and transcriptions, which are also assets, linking back to that source asset via the source_asset_id field. parent_id field is forgotten and unused. 
(The content_type field can be 'source', 'translation', or "transcription'.)
The order_index field preserves the order in which assets should be shown in a quest.
The metadata field is used to store verse or labelling information.

CREATE TABLE public.asset_content_link (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  created_at text NOT NULL DEFAULT CURRENT_TIMESTAMP,
  last_updated text NOT NULL DEFAULT CURRENT_TIMESTAMP,
  asset_id uuid NOT NULL,
  text text,
  active boolean NOT NULL DEFAULT true,
  download_profiles ARRAY,
  source_language_id uuid,
  ingest_batch_id uuid,
  audio jsonb,
  text_search_vector tsvector,
  languoid_id uuid,
  order_index integer NOT NULL DEFAULT 0,
  CONSTRAINT asset_content_link_pkey PRIMARY KEY (id),
  CONSTRAINT asset_content_link_asset_id_fkey FOREIGN KEY (asset_id) REFERENCES public.asset(id),
  CONSTRAINT asset_content_link_languoid_id_fkey FOREIGN KEY (languoid_id) REFERENCES public.languoid(id)
);

asset_content_link records exist so that there can be different expressions of the same unit of language content (i.e., the asset).
So if an asset exists for Mark 2:1, and users need a more dynamic understanding of Mark 2:1, perhaps in multiple Bible translations or even in different languages, an asset_content_link record can exist for each of those expressions.
There may be a better way to do this, perhaps where all these different expressions are individual assets and they're linked to each other with tags corresponding to the type of connection that exists between them. But for now, this is what we have.
This is also why an asset_content_link (instead of an asset) can be linked to a languoid/language (languoid_id).
The audio field has the filename of the audio file held in storage.
The order_index exists for the now-experimental merging feature. If users merge assets together (available in the recording interface), the order in which they're merged is maintained in this field.

CREATE TABLE public.vote (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  asset_id uuid NOT NULL,
  polarity text NOT NULL,
  comment text,
  creator_id uuid,
  active boolean NOT NULL DEFAULT true,
  download_profiles ARRAY,
  ingest_batch_id uuid,
  translation_id uuid,
  CONSTRAINT vote_pkey PRIMARY KEY (id),
  CONSTRAINT votes_creator_id_fkey FOREIGN KEY (creator_id) REFERENCES public.profile(id),
  CONSTRAINT vote_asset_id_fkey FOREIGN KEY (asset_id) REFERENCES public.asset(id)
);

Votes are cast by end users on translations and transcriptions for now (not on source assets). 
The idea is that what is being voted on is not an asset itself, but the quality with which a translation (or transcription) has been made. I.e., how well the end user thinks the meaning of the source content has been transmitted to a different language or modality.

-- Membership subschema

The membership subschema has everything needed (at least minimally) to bring users together into the same project. 
This includes inviting others in, asking for access to a project, linking users to projects (whether as owners or members) and users themselves.

The public.profile records have a direct link to equivalent records in the auth.users table (hard-coded into the supabase auth system). The linked records have the same UUID id value.

The profile_project_link table just connects members/owners with the project(s) they're involved in. 
It's a junction table since one user can have many projects and one project can have many users.

CREATE TABLE public.invite (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  sender_profile_id uuid NOT NULL,
  receiver_profile_id uuid,
  project_id uuid NOT NULL,
  status text NOT NULL CHECK (status = ANY (ARRAY['pending'::text, 'accepted'::text, 'declined'::text, 'withdrawn'::text])),
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  as_owner boolean NOT NULL DEFAULT false,
  email text NOT NULL,
  count integer NOT NULL,
  active boolean NOT NULL DEFAULT true,
  CONSTRAINT invite_pkey PRIMARY KEY (id),
  CONSTRAINT invite_request_project_id_fkey FOREIGN KEY (project_id) REFERENCES public.project(id),
  CONSTRAINT invite_request_receiver_profile_id_fkey FOREIGN KEY (receiver_profile_id) REFERENCES public.profile(id),
  CONSTRAINT invite_request_sender_profile_id_fkey FOREIGN KEY (sender_profile_id) REFERENCES public.profile(id)
);

When an existing owner of a project invites a new member by email, the invite record exists in the db with a blank receiver_profile_id field. 
A trigger is fired with the new record creation to find if the profile exists with tha email, and if so, autopopulate the receiver_profile_id field.
In case the profile doesn't exist yet (i.e., the project owner invited someone who doesn't yet have a langquest account), a trigger also exists on new profile records.
This trigger searches for existing invite records with the same email address (and autopopulate the blank receiver_profile_id field).
Once the record has the receiver_profile_id field populated, sync rules/streams sends the invite record to that recipient where it shows as a notification in the app.
The count field prevents a user from having to repeatedly decline an invite to a project they don't want to be in.

);

Requests are sent by someone wanting access to a project and synced to every owner of that project. 
Once an owner approves the request, it cannot be declined by another owner.

-- Languoid / region subschema

The languoid / region subschema is an adaptation of (and significant deviation of) the glottolog languoid database. 
It also incorporates ideas from Matthew Chua in his own OMT record app.
Basically all the data in it is from glottolog.
The general idea is that we have languages and regions. Languages belong to different families and have different dialects. 
Glottolog uses the term "languoid" to refer to language families, languages, and dialects.
Languoids and regions have different properties which are distinctly separated in this subschema as aliases, sources, and properties.

CREATE TABLE public.project_language_link (
  project_id uuid NOT NULL,

  CONSTRAINT project_language_link_languoid_id_fkey FOREIGN KEY (languoid_id) REFERENCES public.languoid(id)
);

The project_language_link table should techinically be called project_languoid_link, but is from an old version of the schema that had a generic language table. We didn't want to rename a table if we didn't have to.

Languoids can include families, languages, and dialects. The lineage in family trees are captured in the parent_id field.

The languoid_alias table covers all the available names for a given languoid, hopefully in its language and in others.
The subject_languoid_id links to the languoid it is referring to. The label_languoid_id is the language in which the name is being expressed.
E.g., if you want to know the French word for 'Spanish', look up a languoid alias record where subject => Spanish and label => French.
If the label and subject link to the same languoid record, it's an alias_type='endonym' (otherwise 'exonym').

-- download_profiles subschema

Almost every table has a download_profiles column. This is a jsonb array of profile UUIDs indicating which users have "downloaded" (flagged for offline sync) a given record.
When a user downloads a quest or project, the system needs to stamp this array on every related record so that PowerSync's sync rules know what to send to that user's device.
Doing this record-by-record would be expensive, so we precompute "closure" tables that collect all related record IDs in one place, making the download operation a single bulk update.

CREATE TABLE public.project_closure (
  project_id uuid NOT NULL,
  asset_ids jsonb DEFAULT '[]'::jsonb,
  translation_ids jsonb DEFAULT '[]'::jsonb,
  vote_ids jsonb DEFAULT '[]'::jsonb,
  tag_ids jsonb DEFAULT '[]'::jsonb,
  language_ids jsonb DEFAULT '[]'::jsonb,
  quest_ids jsonb DEFAULT '[]'::jsonb,
  quest_asset_link_ids jsonb DEFAULT '[]'::jsonb,
  asset_content_link_ids jsonb DEFAULT '[]'::jsonb,
  quest_tag_link_ids jsonb DEFAULT '[]'::jsonb,
  asset_tag_link_ids jsonb DEFAULT '[]'::jsonb,
  total_quests integer NOT NULL DEFAULT 0,
  total_assets integer NOT NULL DEFAULT 0,
  total_translations integer NOT NULL DEFAULT 0,
  approved_translations integer NOT NULL DEFAULT 0,
  download_profiles ARRAY DEFAULT '{}'::uuid[],
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  source_language_ids jsonb DEFAULT '[]'::jsonb,
  target_language_ids jsonb DEFAULT '[]'::jsonb,
  CONSTRAINT project_closure_pkey PRIMARY KEY (project_id),
  CONSTRAINT project_closure_project_id_fkey FOREIGN KEY (project_id) REFERENCES public.project(id)
);
CREATE TABLE public.quest_closure (
  quest_id uuid NOT NULL,
  project_id uuid NOT NULL,
  asset_ids jsonb DEFAULT '[]'::jsonb,
  translation_ids jsonb DEFAULT '[]'::jsonb,
  vote_ids jsonb DEFAULT '[]'::jsonb,
  tag_ids jsonb DEFAULT '[]'::jsonb,
  language_ids jsonb DEFAULT '[]'::jsonb,
  quest_asset_link_ids jsonb DEFAULT '[]'::jsonb,
  asset_content_link_ids jsonb DEFAULT '[]'::jsonb,
  quest_tag_link_ids jsonb DEFAULT '[]'::jsonb,
  asset_tag_link_ids jsonb DEFAULT '[]'::jsonb,
  total_assets integer NOT NULL DEFAULT 0,
  total_translations integer NOT NULL DEFAULT 0,
  approved_translations integer NOT NULL DEFAULT 0,
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  download_profiles ARRAY DEFAULT '{}'::uuid[],
  source_language_ids jsonb DEFAULT '[]'::jsonb,
  target_language_ids jsonb DEFAULT '[]'::jsonb,
  CONSTRAINT quest_closure_pkey PRIMARY KEY (quest_id),
  CONSTRAINT quest_closure_quest_id_fkey FOREIGN KEY (quest_id) REFERENCES public.quest(id),
  CONSTRAINT quest_closure_project_id_fkey FOREIGN KEY (project_id) REFERENCES public.project(id)
);

The closure tables are precomputed summary rows - one per quest (quest_closure) and one per project (project_closure). 
They store jsonb arrays of every related record ID (assets, translations, votes, tags, languages, and all the junction table IDs like quest_asset_link, asset_content_link, etc.). 
They also store precomputed aggregate counts (total_assets, total_translations, approved_translations) used for progress display in the UI.

quest_closure is kept up to date by Postgres triggers that fire when related data changes:
- A new quest is inserted -> init_quest_closure trigger creates an empty closure row.
- A quest_asset_link is inserted/updated -> trigger appends the asset/link IDs, recomputes language_ids and total_assets.
- A translation is inserted/updated -> trigger finds all affected quests (via quest_asset_link), appends the translation ID, recomputes total_translations and approved_translations.
- A vote is inserted/updated -> trigger traces back through translation -> quest_asset_link to find affected quests, appends the vote ID, recomputes approved_translations.

project_closure rolls up across all quest closures in a project. It additionally has quest_ids and total_quests since it aggregates at the project level.

There are also full-rebuild functions (rebuild_single_quest_closure, rebuild_all_quest_closures, rebuild_single_project_closure) that wipe and recompute a closure from scratch. These exist as a safety net for when closures get stale after bulk operations (like cloning) or for local dev seeding via the populate_closures() utility function.

When a user downloads a quest, the app calls the download_quest_closure RPC, which reads the precomputed ID arrays and stamps the user's profile ID into the download_profiles column of every related record in one shot. download_project_closure does the same at the project level.

-- Cloning subschema

Cloning lets a user duplicate an existing project's structure (quests, assets, content) into a new project, optionally with a different target language and creator. This is how users start translating into a new language from an existing source project.
The cloning process runs server-side as a batched, queue-driven state machine. A Supabase Edge Function (clone-worker) reads jobs from a pgmq queue and repeatedly calls the perform_clone_step RPC, which processes a small batch of records per invocation to avoid timeouts.

The process goes through six stages in order:
  Stage 0 (seed_project): Creates the destination project, copies source language links, adds the target language link, and gives the creator ownership.
  Stage 1 (clone_quests): Copies all quests from the source project into the new project, batch by batch.
  Stage 2 (clone_assets): Copies all active assets that are linked to the source project's quests.
  Stage 3 (clone_acl): Copies asset_content_link records, remapping them to the newly created assets.
  Stage 4 (recreate_links): Recreates the junction tables (quest_asset_link, quest_tag_link, asset_tag_link) using the old-to-new ID mappings so the cloned project has the same structure.
  Stage 5 (recompute_closures): Rebuilds quest_closure rows for the new project in batches, then rebuilds the project_closure once at the end.

During cloning, a session variable (app.clone_mode = 'on') is set so that the normal side-effect triggers (like init_quest_closure) are suppressed - closures are rebuilt from scratch in stage 5 instead.

Each stage is idempotent. If the worker times out or crashes, it picks up where it left off because mapping tables track what's already been copied.

The clone_job table tracks a single cloning operation. 
It stores what project is being cloned (root_project_id), the current status (queued/running/done/failed), options like the new project name and target language, and a progress object that records which stage the job is currently on. 
The clone-worker edge function picks up queued jobs from a pgmq queue and calls perform_clone_step repeatedly until the job is done.

The map_* tables (map_project, map_quest, map_asset, map_acl) are old-to-new ID mapping tables. 
Each row says "for this clone job, source record X became destination record Y." 
They serve two purposes: they make the clone idempotent (if a step is re-run, it skips records that already have a mapping), and they let later stages look up the new IDs when recreating junction tables and links. 
map_acl specifically maps asset_content_link records (acl = asset content link).

CREATE TABLE public.languoid_link_suggestion (
  id uuid NOT NULL DEFAULT gen_random_uuid(),
  languoid_id uuid NOT NULL,
  suggested_languoid_id uuid NOT NULL,
  profile_id uuid NOT NULL,
  match_rank integer NOT NULL DEFAULT 3,
  matched_on text,
  matched_value text,
  status text NOT NULL DEFAULT 'pending'::text,
  active boolean NOT NULL DEFAULT true,
  created_at timestamp with time zone NOT NULL DEFAULT now(),
  last_updated timestamp with time zone NOT NULL DEFAULT now(),
  CONSTRAINT languoid_link_suggestion_pkey PRIMARY KEY (id),
  CONSTRAINT languoid_link_suggestion_languoid_id_fkey FOREIGN KEY (languoid_id) REFERENCES public.languoid(id),
  CONSTRAINT languoid_link_suggestion_suggested_languoid_id_fkey FOREIGN KEY (suggested_languoid_id) REFERENCES public.languoid(id),
  CONSTRAINT languoid_link_suggestion_profile_id_fkey FOREIGN KEY (profile_id) REFERENCES public.profile(id)
);

When users create projects offline, they may create a custom languoid if the one they need isn't synced to their device. 
Once that custom languoid syncs to Postgres, a trigger searches for existing languoids that match by name, alias, or ISO code (using both exact/substring matching and fuzzy trigram similarity). 
Up to 5 matches are inserted as pending suggestion records.
The languoid_id field points to the user's custom languoid, and suggested_languoid_id points to the existing match. 
profile_id is the user who created the custom languoid (and who will see the suggestions). match_rank indicates quality (1=exact, 2=starts-with, 3=contains, 4-6=fuzzy by decreasing similarity). 
matched_on records what was matched ('name', 'alias', or 'iso_code') and matched_value stores the actual string that matched.
When the user accepts a suggestion, an RPC rewrites all references (project_language_link, asset_content_link, profile.ui_languoid_id) from the custom languoid to the suggested one, deactivates the custom languoid, and withdraws all other pending suggestions. 
The user can also dismiss all suggestions to keep their custom languoid.
This feature is behind a toggle in settings and only works online.

Stores exported audio artifacts for quests. 
The intent of this table is to make content available for distribution (or feedback) app owned by other organization(s).
